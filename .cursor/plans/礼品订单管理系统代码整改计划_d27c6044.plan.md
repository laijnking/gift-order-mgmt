# 礼品订单管理系统 — 整改计划

> 版本：v2.1 | 更新日期：2026-05-01
>
> 依据：`docs/review-codex/` 评审文档 + `AGENTS.md` 开发规范

## 问题验证结论

经过代码验证，4份 Codex 文档发现的问题**全部属实**。

**核心问题：规则中心化缺失 + 状态机不是单一事实来源 + 动作没有服务层 + 页面脚本化 + 结果可解释性不足**

---

## 一、AI订单录入模块（`/order-parse`）问题清单

### 1.1 需求-代码不匹配

| # | 问题 | 严重度 | 验证证据 |
|---|------|-------|---------|
| 1 | 商品匹配优先级不一致 | P0 | 文档要求"规格>编码>条码>名称"，代码实际是"编码>条码>规格>名称" |
| 2 | SKU映射查询字段双轨 | P0 | `/api/product-match`用`customer_code`，`/api/order-parse/excel`用`customer_id` |
| 3 | 列映射规则双轨实现 | P0 | `column-mapping-rules.ts`和`excel/route.ts`各有一份`autoDetectColumnMapping` |

### 1.2 代码结构问题

| # | 问题 | 严重度 | 说明 |
|---|------|-------|------|
| 4 | page.tsx过重 | P1 | 同时管理：会话恢复、Excel解析、映射历史、商品匹配、提交录单、结果弹窗 |
| 5 | 规则与流程未分层 | P1 | 规则（映射、匹配）与流程（上传->预览->提交）耦合 |
| 6 | 缺少SSOT守卫 | P1 | 未建立自动化守卫阻断旧字段（如`customer_code`）回流 |
| 7 | 前端与Hook重复实现 | P1 | `use-order-submit.ts`已定义但`page.tsx`未使用，自己实现了一套 |
| 8 | 接口边界重叠 | P2 | `/api/match`与`/api/product-match`都提供商品匹配能力 |

### 1.3 映射保存与复用问题（文档8.1节）

| 问题 | 说明 |
|------|------|
| 映射保存是异步 | 失败时只`console.warn`，用户侧缺少强提示 |
| 指纹匹配严格 | 只有表头完全一致才命中，增删列/改列名会失败 |
| 缺少近似匹配建议 | 指纹不匹配时无提示，用户不知道可以手动复用最近版本 |

### 1.4 导出列名来源问题（文档8.2节）

| 问题 | 说明 |
|------|------|
| 模板覆盖分支 | 无客户映射时退回模板映射，列名可能不是客户原始Excel列名 |
| UI无明确提示 | 用户不知道本次列名来源是"客户映射"还是"模板" |

## 二、订单管理模块（`/orders`）问题清单

### 2.1 功能按钮可用性问题

| # | 问题 | 严重度 | 说明 |
|---|------|-------|------|
| 9 | 批量操作栏禁用规则语义不对称 | P0 | 前三项"必须有选择"，后两项"无选择时不禁用"走默认集合，用户心智不一致 |
| 10 | 按钮缺少disabledReason | P0 | 仅`disabled={boolean}`无提示，用户不知道是"状态不满足"还是"未选中" |
| 11 | 行级操作规则分散 | P1 | 状态可编辑规则存在于`page.tsx`、`use-orders-crud.ts`、`order-edit-dialog.tsx`三处 |
| 12 | 弹窗缺少策略预览 | P1 | 回单弹窗无自动匹配集合预览，派发弹窗无匹配理由详情 |

### 2.2 状态与流转问题

| # | 问题 | 严重度 | 说明 |
|---|------|-------|------|
| 13 | `feedbacked`状态歧义 | P1 | 文档说"已并入回单阶段统计"，代码却是正式状态，存在长期认知偏差 |
| 14 | 状态机不是单一事实来源 | P0 | 状态定义在`lib`，动作约束在页面，字段可编辑性在dialog/hook，三处并存 |
| 15 | 批量动作缺少事务语义 | P1 | 由页面串行PATCH组合，无统一事务语义与统一结果模型 |

### 2.3 API与数据问题

| # | 问题 | 严重度 | 说明 |
|---|------|-------|------|
| 16 | 巨型路由 | P0 | `route.ts` 1423行，GET/POST/PATCH/DELETE混合多种职责 |
| 17 | 选择集使用`Set<Order>` | P1 | 分页/刷新后对象引用变化导致勾选状态可能丢失 |
| 18 | 兼容逻辑侵入主干 | P1 | `transformOrder`混合snake/camel + 新旧字段兜底 |

## 三、发货通知单/物流回单/导出/模板模块问题清单

### 3.1 链路与流程问题

| # | 问题 | 严重度 | 说明 |
|---|------|-------|------|
| 19 | 导出副作用不透明 | P0 | "导出"可能同时触发派发、状态更新、导出记录、文件持久化，用户不清楚 |
| 20 | 回单确认状态不可预测 | P1 | 确认后可能是`returned`或`partial_returned`，用户不易预测结果 |
| 21 | 导出列名来源不明确 | P1 | 用户不知道本次列名来源是"客户映射"还是"模板" |
| 22 | 模板多形态存储 | P2 | `fieldMappings`存在对象格式和数组格式两种形态 |

### 3.2 跨模块共性问题

| 问题 | 说明 |
|------|------|
| 规则源分散 | 状态机、模板优先级、映射优先级散落在多个页面和路由 |
| 双路径实现 | 同一业务能力存在多个API入口，口径容易分叉 |
| 可用性与可执行性不一致 | 按钮可点但后端拒绝的体验风险仍在 |
| 结果可解释性不足 | 缺少"这次为什么是这个结果"的系统性说明 |

## 四、业务决策确认

| 问题 | 决策 |
|------|------|
| 商品匹配优先级 | **以代码实际为准**（编码>条码>规格>名称），更新文档 |
| `feedbacked`状态 | **作为正式状态保留**，更新文档保持一致 |
| `partial_returned`导出 | **分两个选项导出**（已回单/部分回单分开），用户可选择 |

## 五、整改方案

### 阶段 A：SSOT守卫 + 规则收敛（P0，2-3天）

**目标：消除双轨实现，建立唯一规则来源**

#### A.1 建立SSOT守卫

新建 `src/lib/ssot-guard.ts`，禁止在新代码直接使用旧字段查询。

#### A.2 商品匹配优先级统一

新建 `src/lib/order-parse/match-policy.ts`，固化匹配规则：

```typescript
// 匹配优先级（以代码实际为准）
export const MATCH_PRIORITY = ['code', 'barcode', 'spec', 'name'] as const;
```

#### A.3 列映射规则统一

- 移除 `excel/route.ts` 中的本地`autoDetectColumnMapping`
- `excel/route.ts`和`column-mapping-rules.ts`共用`src/lib/column-mapping-rules.ts`的唯一实现
- 增加规则版本号用于审计

#### A.4 SKU映射口径统一

- 统一查询字段为`customer_id`
- `/api/product-match`标记deprecated，过渡期保留

**验收标准：**
- 删除`excel/route.ts`内私有映射规则后，E2E导入行为不变
- `pnpm check:api-contracts` 全部通过

**文档同步：**
- 更新 `docs/Product/模块功能说明.md` - 商品匹配优先级变更
- 更新 `docs/Product/模块交互说明.md` - 如涉及API变更

---

### 阶段 B：状态机配置化 + 按钮矩阵（P0，2-3天）

**目标：状态机成为单一事实来源，按钮权限可解释**

#### B.1 新建状态机配置

新建 `src/lib/order-workflow-config.ts`：

```typescript
export const ORDER_WORKFLOW_CONFIG = {
  allowedTransitions: {
    pending: ['assigned', 'cancelled'],
    assigned: ['notified', 'returned', 'partial_returned', 'cancelled'],
    notified: ['returned', 'partial_returned', 'cancelled'],
    returned: ['feedbacked', 'completed'],
    partial_returned: ['returned', 'feedbacked', 'completed'],
    feedbacked: ['completed'],  // 正式状态，保留
    completed: [],
    cancelled: [],
  },
  editableFieldsByStatus: {
    pending: ['*'],
    assigned: ['remark'],
    notified: ['remark'],
    returned: [],
    partial_returned: [],
    feedbacked: [],
    completed: [],
    cancelled: [],
  },
  bulkActionRules: {
    assign: {
      visible: true,
      enabled: (ctx) => ctx.selectedCount > 0 && ctx.pendingCount > 0,
      disabledReason: {
        noSelection: '请先选择订单',
        noPending: '所选订单中没有待派发状态',
      },
    },
    feedback: {
      visible: true,
      enabled: (ctx) => ctx.selectedCount > 0 && ctx.returnableCount > 0,
      disabledReason: {
        noSelection: '请先选择订单',
        noReturnable: '仅已回单订单可反馈客户',
      },
    },
  },
};
```

#### B.2 按钮矩阵改造

- 批量操作栏：修复"无选择时不禁用"问题，统一语义
- 所有按钮：从`disabled={boolean}`升级为`disabledReason`对象
- 行级操作：统一到`order-workflow-config`驱动

**验收标准：**
- 同一订单在UI可见动作与API实际可执行动作100%一致
- 所有按钮显示disabledReason

**文档同步：**
- 更新 `docs/Product/模块功能说明.md` - 订单状态定义
- 更新 `docs/Product/数据字典.md` - 如涉及状态字段变更

---

### 阶段 C：页面瘦身 + Hook复用（P1，2-3天）

**目标：page.tsx只做编排，逻辑下沉Hook/Service**

#### C.1 订单录入页改造

1. `page.tsx`使用`use-order-submit.ts`替代直接调用
2. Excel处理统一走`use-excel-parser`+服务端解析
3. 映射保存结果改为可见成功/失败状态（不只是console.warn）
4. 增加"手动保存映射"按钮

#### C.2 订单管理页改造

1. 动作下沉到`useOrdersActions`
2. 选择集改为`Set<string>`
3. 批量动作统一走执行器

**验收标准：**
- page.tsx行数减少50%以上
- `pnpm check:api-contracts` 全部通过

---

### 阶段 D：API拆分 + 动作服务化（P0，2-3天）

**目标：路由职责单一，动作有事务语义**

#### D.1 路由拆分

保留`src/app/api/orders/route.ts`为兼容代理，新建：

```
src/app/api/orders/
├── route.ts              # 兼容代理
├── list/route.ts        # GET - 查询列表
├── create/route.ts      # POST - 创建订单
├── [id]/route.ts      # PATCH/DELETE - 单条操作
└── actions/
    ├── bulk-return/route.ts
    ├── bulk-feedback/route.ts
    ├── bulk-complete/route.ts
    └── bulk-delete/route.ts
```

#### D.2 统一动作响应结构

```typescript
interface BulkActionResult {
  successCount: number;
  skippedCount: number;
  failed: Array<{ id: string; reason: string }>;
}
```

#### D.3 导出链路口径修复

拆分导出选项：

```
/api/export-feedback/returned-batch    # 仅 returned 状态
/api/export-feedback/partial-batch       # 仅 partial_returned 状态
```

UI 提供两个导出选项：
- "导出已回单订单" → returned-batch
- "导出部分回单订单" → partial-batch

**验收标准：**
- 批量动作失败原因可逐条追踪
- `pnpm check:api-contracts` 全部通过

**文档同步：**
- 更新 `docs/Product/模块交互说明.md` - API端点变更

### 阶段 F：可解释性 + 质量守卫（P1，1-2天）

**目标：用户理解系统行为，有自动化回归保护**

#### F.1 导出可解释性

- 相关测试可稳定复现并阻断回归

## 七、风险控制

1. **每阶段独立验证**：每阶段完成后运行所有 check 命令，确保不破坏功能
2. **保留向后兼容**：API 拆分阶段保留原路由代理，渐进式迁移
3. **灰度观察**：状态修复先在测试环境验证
4. **禁止大改**：每 PR 只做一个关注点，不混合多个改动
5. **证据先于论断**：所有声明必须附验证命令的实际输出
