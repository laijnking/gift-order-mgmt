# 礼品订单管理系统 — AI订单录入模块深度分析与整改方案

> 版本：v1.0 | 更新日期：2026-04-30 | 更新内容：完成订单录入模块“需求与代码不匹配、功能重复、落地链路不一致”的专项审计，给出分阶段整改方案与验收指标。

## 1. 执行摘要

当前 `/order-parse` 已有可用雏形，但存在三个结构性问题：

1. **规则源分散**：同一能力（列映射、商品匹配）在多个文件重复实现，规则口径不一致。
2. **数据口径漂移**：部分接口仍用旧字段（如 `customer_code`），与当前“`customer_id` 为主”的设计不一致。
3. **前后端职责重叠**：页面层承担过多编排与业务逻辑，导致重复调用、状态复杂、难测试。

结论：目前问题不是“单点 Bug”，而是“规则中心化缺失 + 迁移中间态长期化”。建议采用“先收敛规则，再收敛流程，最后做自动化回归”的三阶段治理。

---

## 2. 需求-代码不匹配清单（重点）

### 2.1 商品匹配优先级与文档不一致

文档要求：**规格 > 编码 > 条码 > 名称**。但 `order-parse/excel` 中的 `matchOrder` 实际先尝试 `code`，再 `barcode`，后 `spec`，再 `name`，顺序不一致。该差异会直接影响自动匹配命中与成本推荐结果。

影响：同一份订单在不同入口（或未来重构后）可能出现不同匹配结果，造成“看起来随机”的业务反馈。

---

### 2.2 SKU映射查询字段口径不一致（新旧模型混用）

`/api/product-match` 按 `product_mappings.customer_code` 查询；而 `order-parse/excel` 已按 `customers.code -> customers.id -> product_mappings.customer_id` 查询，二者并存。

影响：
- 同一客户在不同接口下可能“一个能命中，一个命不中”；
- 会放大“历史数据兼容”复杂度，长期阻塞字段收敛。

---

### 2.3 列映射规则重复实现，存在双轨配置

`autoDetectColumnMapping` 在两个位置重复定义：
- `src/lib/column-mapping-rules.ts`
- `src/app/api/order-parse/excel/route.ts`

且 `route.ts` 内含本地 `CHINESE_COLUMN_MAPPING` 常量，形成“库规则 + 路由私有规则”双轨。

影响：新增别名或修正规则时容易漏改，线上表现不一致。

---

### 2.4 前端页面与 Hook 存在明显重复实现

`order-parse/page.tsx` 仍直接调用 `/api/orders` 并自动保存映射；而 `hooks/use-order-submit.ts` 已实现同类提交流程（同样请求 `/api/orders` + 自动保存映射）。

影响：
- 维护成本翻倍（改一处忘一处）；
- 行为细节（错误提示、统计字段、回调时机）可能逐步分叉。

---

### 2.5 接口边界重叠：`/api/match` 与 `/api/product-match`

两个接口都提供商品匹配能力，职责接近但实现细节不同，且页面侧调用 `/api/match` 多次（批量+单条刷新）。

影响：
- 业务方难以界定“哪个是标准接口”；
- 结果口径、性能和鉴权治理难统一。

---

## 3. 代码结构问题（导致“经常重复功能”的根因）

### 3.1 `page.tsx` 过重（状态机与业务编排耦合）

`page.tsx` 同时管理：会话恢复、Excel解析、映射历史、商品匹配、提交录单、结果弹窗等。虽然目录中已拆出 `hooks/` 和 `components/`，但核心流程仍在页面层内联，造成“名义模块化、实际集中式脚本”。

### 3.2 规则与流程未分层

当前“规则（字段映射、匹配优先级）”与“流程（上传->预览->提交）”耦合在路由和页面中；应由规则库统一输出“可执行策略”，页面只做编排。

### 3.3 缺少“同一事实来源（SSOT）”守卫

项目原则明确“API-Schema 对齐”，但未建立自动化守卫去阻断旧字段回流（如 `customer_code` 再次进入新代码）。

---

## 4. 整改方案（分阶段）

## 阶段A：规则收敛（1~2天，低风险高收益）

1. **建立唯一匹配策略模块**（如 `src/lib/order-parse/match-policy.ts`）：
   - 固化优先级（按最终业务确认）；
   - 暴露统一方法给 `/api/match` 与 `/api/product-match`。
2. **移除路由内重复列映射定义**：
   - `order-parse/excel/route.ts` 改为仅依赖 `column-mapping-rules.ts`；
   - 增加规则版本号（用于审计）。
3. **字段口径统一**：
   - 映射查询统一走 `customer_id`；
   - 旧字段仅在兼容层读取，不允许新增业务逻辑直接依赖。

**验收标准**：
- 同一输入在两个匹配接口返回相同 `matchType` 与 `productId`；
- 删除 `route.ts` 内私有映射规则后，E2E 导入行为不变。

## 阶段B：流程收敛（2~4天，中风险）

1. **页面瘦身**：
   - `page.tsx` 仅保留 UI 组合和状态展示；
   - 提交逻辑统一走 `use-order-submit`；
   - Excel 处理统一走 `use-excel-parser` + 服务端解析。
2. **合并重复 API 能力**：
   - 明确 `/api/match` 为唯一商品/发货方推荐入口；
   - `/api/product-match` 标记 deprecated（保留过渡期并记录调用来源）。
3. **统一错误模型**：
   - 返回结构统一 `{ success, errorCode, message, data }`。

**验收标准**：
- 页面提交链路只保留一套实现；
- 相同异常在 UI 上提示一致。

## 阶段C：质量守卫（1~2天，持续收益）

1. 新增契约测试：
   - 匹配优先级回归（spec/code/barcode/name）；
   - 客户字段兼容回归（新旧字段）。
2. 新增静态检查：
   - 禁止在新代码直接查询 `product_mappings.customer_code`。
3. 将“录单主链路”纳入无人值守验收基线：
   - `check:api-contracts`、`check:business-smoke` 必须覆盖 Excel 导入场景。

**验收标准**：
- 相关 check 脚本可稳定复现并阻断回归。

---

## 5. 建议的最小改造顺序（避免大手术）

1. 先做 **规则收敛**（不改 UI，不改表结构）；
2. 再做 **页面提交流程去重**（只替换调用路径）；
3. 最后做 **接口合并与下线计划**（带监控开关）。

这样可以把风险控制在“可回滚、可观测、可对账”的范围内，避免一次性重构导致业务停摆。

---

## 6. 里程碑与交付物

- M1（本周）：完成规则收敛 PR（策略统一 + 字段口径统一）。
- M2（下周）：完成页面流程收敛 PR（去除重复提交逻辑）。
- M3（次周）：完成质量守卫 PR（契约测试 + 静态规则 + 文档回写）。

每个里程碑均输出：
1. 变更说明；
2. 影响范围；
3. 回滚方案；
4. 验证命令及结果。

---

## 7. 风险提示

- 如果立即“全量重构 order-parse 页面”，高概率影响一线录单效率。
- 若不先统一规则，再加功能，会继续堆积分叉逻辑，后续成本指数上升。

建议：按上面三阶段推进，优先做“规则中心化”和“字段口径清理”。

## 8. 补充深查（针对评审问题）

### 8.1 Excel 字段映射是否会保存？下次同客户能否免手工映射？

**结论：会保存，但“自动复用”依赖条件，当前实现并非绝对免手工。**

1. **保存时机**：订单提交成功后，前端会异步调用 `POST /api/column-mappings` 自动保存本次 `columnMapping`、`headerRow`、`sourceHeaders`。  
2. **保存内容**：后端会写入 `mapping_config`、`source_headers`、`header_fingerprint`、`template_signature`、`feedback_export_headers`，并将旧版本 `is_active=false`，新版本设为 active。  
3. **下次自动加载机制**：前端会按“客户 + 当前表头指纹”请求 `/api/column-mappings/history?...&fingerprint=...`，命中后自动套用映射。  
4. **关键限制**：
   - 只有“表头标准化后完全一致”才命中；客户同一模板若增删列/改列名，自动命中会失败；
   - 映射是“提交后保存”，仅完成解析但未提交订单不会落库；
   - 自动保存是异步调用，失败时只 `console.warn`，用户侧缺少强提示。

**建议补丁**：
- 提交后映射保存改为“可见成功/失败状态”；
- 增加“手动保存映射”按钮，避免仅依赖提交成功触发；
- 指纹匹配失败时增加“近似匹配建议”（同客户最近版本 + 差异列提示）。

### 8.2 客户反馈导出是否直接使用映射表？列名是否保留 Excel 原名？

**结论：是，主路径已按该设计落地，但存在“模板覆盖”分支需要业务确认。**

1. **主路径（有客户映射）**：
   - 导出接口会查询 `column_mappings` 的 active 版本；
   - 优先读取 `feedback_export_headers`（结构：`客户原始列名 -> 系统字段key`）；
   - 生成行数据时按该映射输出列名，并额外追加“快递公司/运单号”。
2. **字段取值来源**：系统从订单主表与 `orders.items` 组装 context（如 `orderNo/customerOrderNo/productName/productCode/productSpec/receiver...`），再按 `feedback_export_headers` 的系统字段 key 取值。
3. **覆盖分支**：若当前客户没有可用 `column_mappings`，系统会退回模板映射（default/explicit/first），此时列名不一定是客户原始Excel列名。

**建议补丁**：
- 在导出 UI 明确展示“本次列来源：客户导入映射 / 模板映射”；
- 对 `column_mapping` 缺失客户给出阻断或二次确认，避免用户误以为一定按原列名导出；
- 对 `feedback_export_headers` 做完整性校验（关键列缺失时提示修复映射）。
