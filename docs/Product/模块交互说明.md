## 2. 详细交互说明

### 2.1 订单录入 → 档案管理

#### 2.1.1 选择客户 → 自动联动业务员/跟单员

**交互场景**：选择客户后，自动按 ID 填充客户档案中绑定的业务员和跟单员。

**交互流程**：

```
1. 用户选择客户
2. 系统查询 customers 表，获取：
   - salesperson_id, salesperson_name: 来自 customers 表的绑定关系（而非按姓名匹配）
   - order_taker_id, order_taker_name: 同上
3. 订单录入表单自动填充对应字段
```

> **设计原则**：优先使用 `salesperson_id` / `order_taker_id`，仅在 ID 缺失时才进入按姓名匹配的兼容分支。

**API调用**：

```
GET /api/customers/[id]
```

#### 2.1.2 Excel 解析 → 列映射保存

**交互场景**：Excel 解析完成后，保存映射时同时写入 `feedback_export_headers`。

**交互流程**：

```
1. 用户上传 Excel，选择客户
2. 系统自动加载历史映射（按表头指纹 SHA256 精确匹配）
3. 用户可手动调整映射
4. 保存映射时：
   - 写入 mapping_config（列索引 → 系统字段名）
   - 写入 source_headers（原始表头数组）
   - 写入 header_fingerprint（SHA256 前 16 位）
   - 写入 template_signature（映射 + 表头行 + 指纹）
   - 写入 feedback_export_headers（"客户列名" → "系统字段名"）
```

**feedback_export_headers 示例**：

```json
{
  "货号": "product_code",
  "品名": "product_name",
  "规格": "product_spec",
  "数量": "quantity"
}
```

### 2.2 订单录入 → 库存管理

#### 2.2.1 商品匹配后查询库存

**交互场景**：商品匹配成功后，查询该商品在各发货方的库存。

**交互流程**：

```
1. 商品匹配成功后，获取 productId
2. 查询 stocks 表，获取所有有库存的发货方
3. 按发货方分组，返回库存信息
4. 如果库存 ≤ 2 台，触发尾货预警
```

### 2.3 订单提交 → 订单中心

#### 2.3.1 订单提交

**交互流程**：

```
1. 用户确认订单信息，点击提交
2. 系统校验：
   - 必填字段：收货人、收货电话、收货地址
   - 商品信息：至少有一条商品明细
3. 生成系统订单号：SYS-YYYYMMDD-XXXX-TIMESTAMP
4. 写入 orders 表（items 为 JSONB 数组）
5. 生成导入批次号（import_batch）
6. 返回提交结果（含创建数量、跳过数量、系统订单号）
```

**重复导入识别**：

```
同批重复：同一批请求中相同客户订单号只保留第一条
系统已存在：同客户同订单号在系统中已存在时跳过
```

**API调用**：

```
POST /api/orders
Content-Type: application/json
Body: {
  orderNo: "KHDD00000001",          // 客户订单号（必填）
  customerCode: "CUST001",          // 客户编码（必填，不存在时填 "UNKNOWN"）
  customerName: "某某公司",          // 客户名称
  salespersonId: "uuid",            // 业务员ID
  salespersonName: "张三",          // 业务员姓名（冗余）
  operatorId: "uuid",               // 跟单员ID
  operatorName: "李四",             // 跟单员姓名（冗余）
  supplierId: "uuid",               // 发货方ID
  supplierName: "京东仓库",          // 发货方名称（冗余）
  items: [{                          // 商品明细数组（必填，至少一条）
    orderNo: "KHDD00000001",        // 订单号（每条 item 内部需携带）
    productName: "苏泊尔手持式吸尘器", // 商品名称（必填，camelCase）
    productCode: "SKU001",          // 商品编码
    productSpec: "03S57-AP",        // 规格型号
    productBrand: "苏泊尔",          // 品牌
    quantity: 1                     // 数量
  }],
  receiver: {                        // 收货信息（必填）
    name: "王五",                    // 收货人姓名
    phone: "13800138000",           // 联系电话
    address: "北京市朝阳区xxx路xxx号", // 收货地址
    province: "北京"                 // 省份
  },
  expressRequirement: "顺丰优先",     // 快递要求
  remark: "加急处理",                // 备注
  source: "manual"                   // 来源标识（manual=手动创建, ai_parse=AI解析, excel=Excel导入）
}
```

> **注意**：items 数组中每个元素的字段使用 **camelCase** 命名（`productName`、`productCode`、`productSpec`、`productBrand`、`orderNo`），与数据库 JSONB 中存储的 `product_name`（snake_case）不同。API 层负责格式转换。

#### 2.3.2 订单号生成规则

**格式**：`SYS-{日期}-{序号}-{时间戳随机码}`

**示例**：`SYS-20260416-0001-M1ABC2D3`

**说明**：
- 日期：YYYYMMDD 格式
- 序号：当日订单从 0001 开始递增
- 时间戳随机码：确保高并发下唯一性

### 2.4 订单派发 → 发货通知单

#### 2.4.1 三种派发模式

发货通知导出支持三种语义完全不同的派发模式，通过 `dispatchMode` 参数区分：

| dispatchMode | 效果 | 副作用 |
|-------------|------|--------|
| `preview` | 仅生成预览内容 | 无任何副作用 |
| `dispatch_only` | 派发订单 | 扣减库存、写入派发记录、写入成本记录 |
| `dispatch_with_persistence` | 派发 + 留痕 | 包含上述所有 + 写入导出记录、文件持久化 |

**dispatch_only 场景**：用户只需要派发订单，不需要文件记录。

**dispatch_with_persistence 场景**：标准发货流程，需要留痕和下载。

#### 2.4.2 按发货方独立解析模板

**交互流程**：

```
1. 用户选择要派发的订单，点击导出
2. 用户选择派发模式（预览 / 仅派发 / 派发并留痕）
3. 对每个发货方循环：
   a. resolvePreferredTemplate(targetType='supplier', targetId=supplierId)
   b. 若有发货方专属模板 → 使用专属模板，商品字段按 supplierProductCode 优先
   c. 若无专属模板 → fallback 到默认模板，商品字段按 productCode 优先
   d. 若无任何模板 → 使用内置默认字段
   e. 生成 dispatchItems（含发货方商品编码快照）
4. 执行对应副作用（按派发模式）
5. 返回 dispatchSummary（newDispatchCount / reusedDispatchCount / assignedOnlyCount）
```

#### 2.4.3 发货方商品编码快照

**交互场景**：派发时，通过 `findSupplierProductMapping` 查询 `product_mappings` 表获取发货方商品编码/名称/规格，快照写入 `dispatch_records.items`。

**数据来源**：`product_mappings` 表的 `supplier_product_code`、`supplier_product_name`、`supplier_product_spec` 字段，通过 `(supplier_id, product_id)` 联合查询。

**交互流程**：

```
1. 订单派发时，根据 orders.items 中的 product_id 和 supplier_id
2. 调用 findSupplierProductMapping(client, supplierId, productId)
3. 查询 product_mappings 表：SELECT supplier_product_code, supplier_product_name, supplier_product_spec
   WHERE supplier_id = $1 AND product_id = $2
4. 若查询到映射 → 使用映射中的 supplierProductCode/Name/Spec
   若无映射 → fallback 到 stocks 表的 product_code / product_name
5. 写入 dispatch_records.items[]:
   {
     supplierProductCode: mapping?.supplierProductCode || stock.product_code,
     supplierProductName: mapping?.supplierProductName || stock.product_name,
     supplierProductSpec: mapping?.supplierProductSpec || stock.supplier_product_spec,
     productCode: stock.product_code,   // 系统商品编码（用于内部成本核算）
     productName: stock.product_name,
     ...
   }
6. 导出时直接从 dispatch_items 读取，不再查询 product_mappings
```

**fallback 规则**：

| 字段 | 有 product_mappings 映射 | 无映射时的 fallback |
|------|------------------------|------------------|
| supplierProductCode | `product_mappings.supplier_product_code` | `stocks.product_code`（系统商品编码） |
| supplierProductName | `product_mappings.supplier_product_name` | `stocks.product_name`（系统商品名称） |
| supplierProductSpec | `product_mappings.supplier_product_spec` | `stocks.supplier_product_spec` |

**好处**：
- 导出时无需跨表查询，性能更好
- 快照稳定，导出内容不受后续 mapping 变更影响
- 订单详情页可展示派发时匹配的发货方商品信息

#### 2.4.4 库存容错机制（允许负库存发货）

**业务规则**：发货通知单导出不因库存不足而阻断。允许负库存发货，确保业务连续性。

**交互流程**：

```
1. 派发时查找发货方库存（findStockForItem）
2. 若找到匹配库存 → 正常扣减，库存不足时照常扣减（允许变为负值）
3. 若未找到匹配库存 → 不阻断派发，使用订单商品信息兜底：
   - supplierProductCode = item.cu_product_code || item.product_code
   - supplierProductName = item.cu_product_name || item.product_name
   - 记录 stockWarning：「商品名」未找到发货方库存，已按负库存发货
4. 库存不足时 → 记录 stockWarning：「商品名」库存不足（当前 X 台，需要 Y 台），已按负库存发货
5. 所有 warning 收集后返回前端，在结果对话框中以琥珀色警告列表展示
```

**场景对照**：

| 场景 | 旧行为 | 新行为 |
|------|--------|--------|
| 无匹配库存 | 抛出错误，阻断该订单派发 | 允许派发，记录 warning，用订单商品信息兜底 |
| 库存不足 | 抛出错误，阻断该订单派发 | 允许派发（扣减为负），记录 warning |
| 正常库存 | 正常扣减 | 不变 |

**前端提示**：
- 仅有 warning 时：toast.warning 提示「导出完成：X 个发货方，共 Y 个订单（N 条库存警告）」
- 有 error 时：toast.error 提示，结果对话框展示错误列表（红色）+ 警告列表（琥珀色）
- API 响应新增 `warnings` 字段（string[]），与 `errors` 同级返回

#### 2.4.4 派发副作用幂等

**交互场景**：同一订单因状态漂移重新落回 `pending` 后再次派发。

**处理方式**：

```
1. 查询该订单是否已有 dispatch_records
2. 若有且订单状态为 assigned+ → 复用既有派发记录
   - 不再扣减库存
   - 不再写派发记录
   - 不再写成本记录
3. 若无 → 执行完整派发流程
```

#### 2.4.5 库存扣减

**触发时机**：dispatch_only / dispatch_with_persistence 模式派发时

**交互流程**：

```
1. 订单派发后，更新 stocks 表
   - quantity = quantity - order_quantity
2. 写入 stock_versions 表记录变更
   - change_type: "order"
   - change_quantity: -order_quantity
3. 如果库存扣减到 ≤ 2 台，触发预警（通过预警执行器）
```

### 2.5 发货通知单 → 物流回单

#### 2.5.1 发货方回传

**交互场景**：发货方发货后，通过 Excel 回传快递单号。

**交互流程**：

```
1. 用户上传回单 Excel（包含订单号、快递公司、快递单号）
2. 系统按匹配规则匹配订单：
   - 优先级1：快递单号精确匹配
   - 优先级2：订单号 + 收货人手机号匹配
   - 优先级3：订单号 + 收货人姓名匹配
   - 优先级4：收货人手机号 + 商品名称匹配
3. 匹配结果：
   - 单一匹配 → 自动匹配成功（auto_matched）
   - 多候选匹配 → 冲突（conflict），进入复核池
   - 无匹配 → 未匹配（unmatched），进入复核池
4. 用户可在复核池中处理冲突/未匹配记录
5. 批量确认后更新订单状态
```

#### 2.5.2 冲突检测机制

**触发条件**：同一回单命中多个不同订单候选。

**处理方式**：

```
1. 自动匹配接口返回 conflictCount > 0
2. 页面自动切到"冲突"筛选 tab
3. 用户点击"处理冲突"，查看候选订单列表
4. 候选按"更像当前回单的结果"优先排序
5. 用户选择正确订单，点击确认
6. 系统执行手动匹配（manual_matched），更新订单物流信息
```

**API调用**：

```
POST /api/return-receipts/confirm
Body: {
  recordId: "uuid",
  orderId: "uuid",
  action: "resolve_conflict"
}
```

### 2.6 回单 → 历史成本库

#### 2.6.1 回单后更新成本信息

**交互场景**：回单确认后，更新历史成本库的快递信息。

**交互流程**：

```
1. 回单匹配成功后（auto_matched / manual_matched）
2. 查询 order_cost_history 表
3. 更新：
   - express_company: 快递公司
   - tracking_no: 快递单号
   - returned_date: 回单日期
   - total_amount: total_cost + express_fee + other_fee
4. 更新 orders.returned_at（回单时效报表依赖此字段）
```

**数据库操作**：

```sql
UPDATE order_cost_history SET
  express_company = $1,
  tracking_no = $2,
  returned_date = CURRENT_DATE,
  total_amount = total_cost + COALESCE(express_fee, 0) + COALESCE(other_fee, 0),
  updated_at = NOW()
WHERE order_id = $3;

UPDATE orders SET
  returned_at = NOW(),
  updated_at = NOW()
WHERE id = $3;
```

### 2.9 库存 → 预警中心

#### 2.9.1 预警执行器三阶段

预警执行器（`src/lib/alert-executor.ts`）统一管理所有预警规则，执行后自动处理三类状态：

```
阶段1：新增
  条件：满足规则条件 + 无未处理预警
  操作：创建新 alert_records

阶段2：复用
  条件：满足规则条件 + 已有未处理预警
  操作：不做任何写操作（预警已存在）

阶段3：自动关闭
  条件：不满足规则条件 + 有未处理预警
  操作：更新 is_resolved=true, resolved_at=NOW()
```

#### 2.9.2 三类规则

| 规则 | 规则编码 | 扫描对象 | 触发条件 |
|------|----------|----------|----------|
| 库存不足 | `LOW_STOCK_ALERT` | stocks | quantity ≤ threshold |
| 待派发超时 | `ORDER_TIMEOUT_ALERT` | orders(status=pending) | created_at 距今超过 timeout_hours |
| 回单超时 | `RETURN_DELAY_ALERT` | orders(status=assigned) | assigned_at 距今超过 delay_hours |

#### 2.9.3 触发时机

- **手动触发**：用户在预警中心页面点击"执行全部规则"或"执行规则"按钮
- **API调用**：`POST /api/alert-rules/execute`

## 5. 异常处理

### 5.1 订单提交异常

| 异常场景 | 处理方式 |
|----------|----------|
| 订单号重复 | 使用时间戳+随机码重新生成 |
| 库存不足 | 提示用户选择其他发货方（派发时检查） |
| 客户不存在 | 提示先创建客户档案 |
| 商品未匹配 | 允许提交为"未匹配"状态 |
| 同批重复 | 保留第一条，跳过后续，返回跳过数量和原因 |
| 系统已存在重复 | 跳过整条，返回已存在系统单号 |

### 5.2 回单匹配异常

| 异常场景 | 处理方式 |
|----------|----------|
| 订单不存在 | 记录为"未匹配"，进入复核池 |
| 快递单号重复 | 提示用户检查数据 |
| 订单已回单 | 跳过或提示重复回单 |
| 多候选冲突 | 标记为 conflict，进入复核池 |
| 重复导入 | 检测到已存在记录后跳过，返回 duplicateSummary |

### 5.3 库存异常

| 异常场景 | 处理方式 |
|----------|----------|
| 库存为负数 | 禁止派发，提示补货 |
| 库存不足 | 提示库存不足或允许超卖（通过预警执行器提前预警） |
| 撞单 | 通过尾货预警（库存 ≤ 2 台）提前告知 |

### 5.4 导出异常

| 异常场景 | 处理方式 |
|----------|----------|
| 文件未持久化 | 降级为"按需重新生成"下载模式 |
| 导出记录不存在 | 返回 404 错误 |
| 非法 detailIndex | 返回明确错误，不回退到整批 ZIP 下载 |
| S3 不可用 | 降级到本地磁盘模式（若有 fallback 配置） |

```
│  │ 4. XLSX 生成单个 .xlsx 文件                                       │    │
│  │ 5. 企微 media/upload API 上传 → 获得 media_id                     │    │
│  │ 6. 企微 message/send API 发送到群                                  │    │
│  │ 7. INSERT INTO wecom_feedback_tasks（记录，防止重复发送）           │    │
│  └──────────────────────────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────────────────────────┘
```
