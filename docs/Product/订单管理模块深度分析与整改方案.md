# 礼品订单管理系统 — 订单管理模块深度分析与整改方案

> 版本：v1.0 | 更新日期：2026-05-01 | 更新内容：对 `/orders` 模块进行全方位审计（含功能按钮可用性、状态流转一致性、前后端职责边界、可维护性与验收方案）。

## 1. 执行摘要

当前订单管理模块“能用”，但存在明显的**产品语义与工程实现双重债务**：

1. **状态口径未彻底收敛**：`feedbacked` 在代码中仍是真实状态节点，和“回单阶段聚合口径”容易混淆。
2. **页面编排过重**：`orders/page.tsx` 聚合了筛选、批量动作、状态流转、导出、回单、派发弹窗编排。
3. **单路由职责过大**：`/api/orders/route.ts` 承担查询、创建、更新、删除、批量删除、转换、权限、去重等大量能力。
4. **按钮可用性规则分散**：按钮启停逻辑分布于 page + hook + dialog，多处口径稍有不一致即出现“按钮能点但动作失败”。

结论：问题本质是**缺少状态机中心 + 动作服务层 + 统一按钮权限矩阵**。

---

## 2. 全面问题盘点

## 2.1 状态模型与业务口径

- `src/lib/order-status.ts` 中将 `feedbacked` 定义为正式状态并参与标签/颜色/选项；
- 页面上存在 `returned -> feedbacked -> completed` 批量动作链路；
- 这会让“统计口径中的回单阶段聚合”与“真实状态节点”产生长期认知偏差。

**风险**：报表、导出、自动化任务在状态判断上容易各自实现，最终口径漂移。

## 2.2 页面职责过重（维护成本高）

`orders/page.tsx` 目前承担：
- 数据拉取与分页；
- 筛选条件与批次摘要；
- 单条/批量按钮动作；
- 多类弹窗与结果回写；
- 智能匹配与手动派发。

虽然已有 hooks/components 拆分，但关键编排仍集中在页面，属于“组件化外观、脚本化内核”。

## 2.3 API 结构问题（巨型路由）

`/api/orders/route.ts` 同时覆盖多个职责：
- GET 列表查询+权限范围过滤；
- POST 创建（含重复校验、编号生成、商品字段整形）；
- PATCH 更新（状态变更与字段兼容）；
- DELETE 单删/批删。

**风险**：任何改动都可能触发大半径回归；测试不易做细粒度隔离。

## 2.4 数据兼容逻辑侵入业务主干

- `transformOrder` 及创建/更新路径同时处理 snake_case/camelCase 和新旧字段兜底；
- 兼容逻辑没有隔离在 adapter 层，导致业务代码可读性下降。

## 2.5 选择集设计风险

当前列表勾选使用 `Set<Order对象>`，依赖对象引用稳定性。分页刷新/重拉数据后对象实例变化，容易出现“选中状态丢失或错位”。

建议统一为 `Set<string>`（orderId）。

## 3. 功能按钮可用性深度审计（重点）

## 3.1 批量操作栏（BulkActionBar）

### 现状

- 分派发货方：`disabled={!hasSelection || pendingCount === 0}`
- 发货通知：`disabled={!hasSelection || assignedCount === 0}`
- 物流回单：`disabled={!hasSelection || assignedCount === 0}`
- 反馈给客户：`disabled={hasSelection && returnableCount === 0}`
- 导出金蝶：`disabled={hasSelection && feedbackedCount === 0}`

### 问题

1. **禁用规则语义不对称**：
   - 前三项是“必须有选择”；后两项在“无选择”时并不禁用（会走默认集合逻辑），用户心智不一致。
2. **按钮文案与动作边界不透明**：
   - 例如“发货通知”按钮并非直接执行通知，而是跳转/打开其他页面链路；用户预期可能是“一键完成”。
3. **缺少禁用原因可视化**：
   - 仅 disabled 无提示，用户不知道是“状态不满足”还是“未选中”。

## 3.2 行级操作按钮

### 现状

- 查看：始终可用
- 编辑：`canEditOrder` 控制
- 删除：`canDeleteOrder` 控制
- 派发：仅 `pending`
- 发货通知单：仅 `assigned && supplierId`

### 问题

1. **行为分支分散**：状态可编辑规则同时存在于 `page.tsx`、`use-orders-crud.ts`、`order-edit-dialog.tsx`。
2. **提示口径可能不一致**：
   - 页面提示“可编辑字段范围”与后端真正允许更新字段未完全由同一配置驱动。
3. **缺少“为什么不可用”的统一入口**：
   - 某些按钮直接不渲染，而非禁用+说明，增加用户理解成本。

## 3.3 弹窗操作按钮

### 物流回单弹窗

- 文案支持“已选订单按顺序匹配”与“未选时自动匹配已派发订单”；
- 但该“自动匹配集合”策略在 UI 上缺少可视化预览，误操作风险较高。

### 派发弹窗

- 支持智能匹配与手动选择；
- 但“推荐/无库存/新商品”策略解释不够结构化，缺少一键查看匹配理由详情。

## 4. 问题根因（架构视角）

1. **状态机不是单一事实来源**：状态定义在 `lib`，动作约束在页面，字段可编辑性在 dialog/hook，三处并存。
2. **动作没有服务层**：批量动作由页面串行 PATCH 组合，缺少统一事务语义与统一结果模型。
3. **按钮权限矩阵未产品化**：可用性规则没有抽象成配置，导致 UI/后端各自硬编码。
4. **兼容逻辑长期滞留业务层**：迁移成本被“日常需求开发”放大。

---

## 5. 订单管理专项整改方案

## 阶段A（P0）：状态机与按钮矩阵统一

1. 新建 `order-workflow-config.ts`：
   - `allowedTransitions`
   - `editableFieldsByStatus`
   - `actionsByStatus`
   - `bulkActionRules`
2. 前端按钮全部从配置渲染：
   - `visible`
   - `enabled`
   - `disabledReason`
3. 后端 PATCH/Bulk API 复用同一状态机校验。

**验收**：同一订单在 UI 可见动作与 API 实际可执行动作 100% 一致。

## 阶段B（P0）：路由拆分 + 动作服务化

1. 拆分 `/api/orders/route.ts` 为查询与动作端点。
2. 引入 `orders-action-service`：统一处理批量回单、反馈、归档、删除。
3. 返回统一结构：`{ successCount, skippedCount, failed: [{id, reason}] }`。

**验收**：批量动作失败原因可逐条追踪，不再只给总失败提示。

## 阶段C（P1）：页面瘦身与选择稳定性治理

1. `selectedOrders: Set<Order>` -> `selectedOrderIds: Set<string>`。
2. 页面仅负责事件分发，动作细节全部下沉 hook/service。
3. “未选中时默认集合”改为显式二次确认并展示目标订单数。

**验收**：翻页/刷新/筛选切换后勾选稳定，批量动作目标集合可预览。

## 阶段D（P1）：兼容层隔离

1. 新建 `order-adapter.ts` 统一 snake/camel 与新旧字段转换。
2. `transformOrder` 与创建/更新路径只调用 adapter。

**验收**：新增字段只需在 adapter 修改一处。

## 6. 按钮可用性目标模型（建议）

每个按钮统一四元组定义：

```ts
{
  action: 'assign' | 'ship_notice' | 'return' | 'feedback' | 'complete' | 'delete',
  visible: (ctx) => boolean,
  enabled: (ctx) => boolean,
  disabledReason: (ctx) => string | null,
}
```

示例（批量反馈按钮）：
- visible: true
- enabled: `selectedCount > 0 && selectedAllIn(['returned'])`
- disabledReason:
  - 未选择订单 -> “请先选择订单”
  - 选中含非 returned -> “仅已回单订单可反馈客户”

这样可以消除“前端按钮可点但后端拒绝”的体验断层。

---

## 7. 验证与验收清单

1. **状态契约测试**：每个状态只允许配置内动作。
2. **按钮矩阵测试（前端）**：不同状态组合下 visible/enabled/reason 快照测试。
3. **批量动作幂等测试**：重复提交不产生脏状态。
4. **权限回归**：`all/department/self` 下操作可见性一致。
5. **业务冒烟**：录单 -> 派发 -> 回单 -> 反馈 -> 归档 全链路。

---

## 8. 优先落地建议（两周计划）

### Week 1
- 完成状态机配置化 + 按钮矩阵接入（不改业务逻辑，仅替换判定来源）。
- 拆出第一个动作端点（例如 `bulk-feedback`）验证方案。

### Week 2
- 完成全部批量动作服务化。
- 完成 `Set<string>` 选择集改造。
- 补齐按钮可用性快照测试与状态契约测试。

---

## 9. 风险提示

- 不建议一次性重写订单页，优先做“判定口径统一”与“动作服务化”。
- 若先做 UI 改版而不先做状态机收敛，问题会被视觉优化掩盖，后续仍反复出现。
