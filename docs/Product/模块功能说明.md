## 1. 系统概述

### 1.1 系统定位

礼品一件代发业务订单管理系统（Order Management System），支持多渠道订单收集、发货方智能匹配、订单自动拆分导出和快递回单管理。

### 1.2 核心业务流程

```
客户 ──(微信/表格)──▶ 跟单员 ──▶ 订单录入 ──▶ 订单中心 ──▶ 匹配发货方 ──▶ 派发订单 ──▶ 发货方回传快递 ──▶ 回单确认 ──▶ 汇总反馈
```

### 1.3 技术架构

| 层级 | 技术选型 |
|------|----------|
| 前端框架 | Next.js 16 (App Router) + React 19 |
| UI组件 | shadcn/ui (基于 Radix UI) |
| 样式方案 | Tailwind CSS 4 |
| 后端框架 | Next.js API Routes |
| 数据库 | Supabase PostgreSQL + RLS策略 |
| 文件解析 | XLSX (Excel 解析) |
| 存储 | 本地磁盘 / S3 兼容对象存储 |

### 1.4 设计原则

- **API-Schema 对齐**：以 API 代码为事实来源，Schema 和数据库跟随 API 实际使用的字段名
- **派发快照**：发货方商品编码在派发时快照写入订单，不再导出时重复查询
- **幂等派发**：同一订单重复派发时复用既有副作用（库存、派发记录、成本）
- **客户列名还原**：客户反馈导出时使用客户导入时的原始列名

| 订单搜索 | 按订单号、收货人、手机号搜索 | P0 |
| 订单详情 | 查看订单完整信息（商品明细、发货方快照） | P0 |
| 状态管理 | 订单状态流转（待派发 → 已派发 → 已回单 → 已完成） | P0 |

### 4.3 订单状态说明

| 状态值 | 显示名称 | 说明 | 可流转状态 |
|--------|----------|------|-----------|
| `pending` | 待派发 | 订单已导入，等待分配发货方 | assigned, cancelled |
| `assigned` | 已派发 | 已分配发货方（内部派单），等待导出发货通知单 | notified, partial_returned, returned, cancelled |
| `notified` | 通知发货 | 已导出发货通知单，通知了发货方 | partial_returned, returned, cancelled |
| `partial_returned` | 部分回单 | 部分商品已回单 | returned, completed |
| `returned` | 已回单 | 发货方已回传快递单号 | completed |
| `completed` | 已完成 | 订单完结 | - |
| `cancelled` | 已取消 | 订单取消 | - |

> **关于 `assigned` vs `notified`**：`assigned`（已派发）是内部操作（将订单派给发货方），`notified`（通知发货）是外部通知（导出发货通知单，通知发货方开始备货）。状态流转：`pending → assigned → notified → partial_returned/returned → completed`。

> **关于 feedbacked**：`feedbacked` 已并入"回单阶段"统计，不作为独立状态存在。首页和报表口径中的"回单阶段"聚合包含 `assigned`（派发中）、`notified`（已通知发货）、`partial_returned`（部分回单）、`returned`（已回单）、`feedbacked`（已反馈）五种状态。

## 5. 订单录入模块

### 5.1 功能说明

订单录入模块支持通过 Excel 导入订单，自动解析表头、智能匹配商品和发货方、历史映射自动加载，减少人工录入工作量。

### 5.2 功能清单

| 功能点 | 说明 | 优先级 |
|--------|------|--------|
| Excel上传 | 支持 .xls/.xlsx 格式，读取整张 sheet | P0 |
| 列映射 | 自动检测列名，支持手动调整，可自动加载历史映射 | P0 |
| 历史映射自动加载 | 按表头指纹（SHA256）精确匹配，自动加载历史配置（默认折叠） | P0 |
| 商品匹配 | 自动匹配系统商品（名称/规格/编码/条码） | P0 |
| 发货方匹配 | 按库存、价格自动推荐发货方 | P0 |
| 尾货预警 | 库存 ≤ 2 台时提醒，防止撞单 | P0 |
| 订单预览 | 解析后预览确认 | P0 |
| 映射诊断 | 展示识别列数、映射命中率、未识别表头、冲突字段 | P0 |
| 结果统计 | 展示创建数量、重复跳过、商品匹配统计 | P0 |
| 按导入批次回看 | 成功后弹窗一键跳转本批待派发订单 | P0 |
| 客户联动 | 选择客户后按 ID 自动填充业务员/跟单员 | P0 |

### 5.3 实现逻辑

#### 5.3.1 Excel 解析流程

```
1. 用户上传 Excel 文件
2. 系统读取整张 sheet 数据（不再只读预览前 20 行）
3. 自动检测列名映射（支持以下字段）：
   - 基础信息：客户订单号、单据日期、单据编号
   - 商品信息：商品名称、商品编码、规格型号、数量、单价
   - 收货信息：收货人、收货电话、收货地址
   - 快递信息：物流公司、快递单号
4. 按订单号分组，生成订单列表
5. 返回解析结果供用户确认

历史映射自动加载流程：
1. 选择客户后，系统根据 Excel 表头计算指纹（SHA256）
2. 查询该客户是否有完全匹配的映射历史（按 header_fingerprint）
3. 若命中，自动加载映射（绿色提示），默认折叠列映射区域
4. 用户手动修改任一列时，自动加载标记取消
5. 保存映射时，自动构建 feedback_export_headers 存入数据库
```

#### 5.3.2 商品匹配算法

```
优先级：规格匹配 > 编码匹配 > 条码匹配 > 名称匹配

1. 规格匹配：客户商品规格 精确匹配 系统商品规格
2. 编码匹配：客户商品编码 精确匹配 系统商品编码
3. 条码匹配：客户商品条码 精确匹配 系统商品条码
4. 名称匹配：客户商品名称 模糊匹配 系统商品名称

匹配结果：
- matched：精确匹配
- fuzzy：模糊匹配（需人工确认）
- unmatched：未匹配（需手动选择）
```

#### 5.3.3 发货方匹配规则

```
匹配条件：
1. 商品在发货方有库存（quantity > 0）
2. 商品与收货地址省份兼容（无快递限制）
3. 按综合成本排序（商品单价 + 预估运费）

优先级：
1. 内部仓库优先（type = 'self'）
2. 成本最优
3. 库存充足（库存 > 订单数量）
```

#### 5.3.4 重复导入识别

```
同批重复：同一批导入请求中，相同客户订单号只保留第一条
系统已存在：同客户同订单号在系统中已存在时跳过
结果弹窗展示：创建数量、跳过数量、重复原因、已存在系统单号
```

发货通知单模块按发货方生成派发单，支持自定义模板和三种派发模式：预览（preview）、仅派发（dispatch_only）、派发+留痕（dispatch_with_persistence）。

| 预览确认 | 预览发货通知内容，确认后执行派发 | P0 |
| 按发货方导出 | 按发货方拆分订单，生成独立 Excel 文件 | P0 |

| 结果弹窗 | 展示派发结果、模板来源、下载入口 | P1 |
| 查看导出记录 | 从结果弹窗跳转导出记录页 | P1 |

### 6.3 三种派发模式

| 模式 | dispatchMode | 效果 |
|------|-------------|------|
| 预览 | `preview` | 生成预览内容，不派发、不扣库存、不写记录 |
| 仅派发 | `dispatch_only` | 派发订单、扣减库存、写入派发记录和成本，不写导出记录 |
| 派发并留痕 | `dispatch_with_persistence` | 执行所有副作用，并写入导出记录和文件持久化 |

### 6.4 发货方专属模板

**匹配优先级**：
1. 命中 `targetType=supplier` + `targetId` 一致的模板
2. 通过 `template_links` 关联
3. fallback 到默认模板 / 第一个可用模板

**商品字段优先级**（在 `rowsForOrder` 中实现）：

| 场景 | productCode 来源 | productName 来源 | productSpec 来源 |
|------|-----------------|-----------------|-----------------|
| 发货方专属模板（从 dispatchItems） | `supplierProductCode` → `productCode` | `supplierProductName` → `productName` | `supplierProductSpec` → `productSpec` |
| 通用模板（从 dispatchItems） | `productCode` → `supplierProductCode` | `productName` → `supplierProductName` | `productSpec` → `supplierProductSpec` |
| 发货方专属模板（无派发记录，走 parseItems） | `cu_product_code` → `product_code` | `cu_product_name` → `product_name` | `cu_product_spec` → `product_spec` |

## 7. 物流回单模块

### 7.1 功能说明

物流回单模块接收发货方回传的快递单号，支持直接上传 Excel 自动匹配订单，无需手动选择发货方。系统根据订单号自动匹配，并支持部分回单场景（同一订单分批回单时）。

### 7.2 功能清单

| 功能点 | 说明 | 优先级 |
|--------|------|--------|
| 回单导入 | 直接上传 Excel（无需选择发货方），系统自动按订单号匹配 | P0 |
| 运费成本 | 支持从 Excel 读取运费成本字段 | P0 |
| 自动匹配 | 按系统订单号精确匹配 > 客户订单号精确匹配 > 模糊匹配 | P0 |
| 冲突检测 | 同一回单命中多个订单候选时标记为冲突 | P0 |
| 复核池 | 查看和处理冲突/待复核的回单 | P0 |
| 部分回单 | 同一订单分批回单时，状态为"部分回单"，物流信息用";"隔开 | P0 |
| 手动匹配 | 人工选择正确订单 | P1 |
| 批量确认 | 批量确认回单，更新订单状态 | P0 |
| 回单查询 | 查询回单处理进度 | P1 |

### 7.3 回单匹配规则

```
匹配优先级：
1. 系统订单号 (sys_order_no) 精确匹配
2. 客户订单号 (order_no) 精确匹配
3. 客户订单号模糊匹配

注意：不再强制依赖发货方筛选，按订单号全局匹配

匹配结果：
- auto_matched：自动匹配成功
- conflict：冲突（同一回单命中多个不同订单候选）
- manual_matched：人工确认匹配
- pending：未匹配

匹配成功后 → 更新 orders tracking_no, express_company, freight_cost, returned_at
- 若订单已有回单记录 → status='partial_returned'（部分回单）
- 若订单无回单记录 → status='returned'（已回单）

冲突 → 进入复核池，等待人工处理
```

### 7.4 部分回单支持

当同一订单分批导入回单时：
- 物流信息（快递公司、快递单号）用 `;` 隔开追加到订单
- 订单状态标记为 `partial_returned`（部分回单）
- 待全部商品回单后，状态更新为 `returned`（已回单）

### 7.5 回单状态机

```
待匹配(pending) ──自动匹配──▶ 自动匹配成功(auto_matched) ──批量确认──▶ 已确认(confirmed)
                  │
                  ├──冲突(conflict)──▶ 人工处理──▶ 手动匹配(manual_matched) ──确认──▶ 已确认(confirmed)
                  │
                  └──未匹配(pending)──▶ 人工处理
```

### 11.1 功能说明

预警中心模块统一管理所有业务预警，包括库存不足预警、待派发超时预警、回单超时预警，支持预警规则配置和预警记录处理。

### 11.2 功能清单

| 功能点 | 说明 | 优先级 |
|--------|------|--------|
| 预警规则配置 | 配置各类预警规则（库存/派发/回单） | P1 |
| 预警规则执行 | 执行全部规则 / 单条执行 | P1 |
| 预警规则启用/禁用 | 控制预警是否生效 | P1 |
| 预警记录列表 | 查看所有预警，支持按规则/状态筛选 | P1 |
| 预警处理 | 标记预警为已处理，记录处理说明 | P1 |
| 预警统计 | 统计预警分布情况 | P1 |

### 11.3 预警类型

| 预警类型 | 触发条件 | 规则编码 |
|----------|----------|----------|
| `low_stock` | 库存 ≤ 阈值（默认 2 台） | `LOW_STOCK_ALERT` |
| `order_timeout` | 待派发超过 N 小时 | `ORDER_TIMEOUT_ALERT` |
| `return_delay` | 回单超过 N 小时未回传 | `RETURN_DELAY_ALERT` |

### 11.4 预警规则执行器机制

预警执行器（`src/lib/alert-executor.ts`）通过 `/api/alert-rules/execute` 统一执行，流程如下：

```
1. 扫描所有启用中的预警规则
2. 对每个规则扫描对应业务对象（stocks / orders）
3. 对仍满足条件且无未处理预警的对象 → 创建新预警记录
4. 对已满足条件且已有未处理预警的对象 → 复用既有预警
5. 对不再满足条件但有未处理预警的对象 → 自动关闭预警（is_resolved=true）
```
