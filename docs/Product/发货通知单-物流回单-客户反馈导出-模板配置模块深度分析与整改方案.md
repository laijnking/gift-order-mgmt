# 礼品订单管理系统 — 发货通知单 / 物流回单 / 客户反馈导出 / 模板配置 深度分析与整改方案

> 版本：v1.0 | 更新日期：2026-05-01 | 更新内容：对四个 P1 核心协同模块做全局链路审计，补齐跨模块口径、按钮可用性、模板治理与执行落地风险分析。

## 1. 全局结论（先看）

这四个模块的问题不是孤立 Bug，而是同一个系统性问题在不同环节的表现：

1. **流程链路长、状态耦合高**：`orders` 状态、`dispatch_records`、`return_receipts`、`export_records`、`templates` 多表协同，任何口径偏差都会放大。
2. **“模板/映射/默认值”三套策略并存**：专属模板、默认模板、列映射回放同时存在，分支行为复杂。
3. **按钮动作“可点 ≠ 可执行”风险仍在**：前端可用性判断与后端约束并未完全由同一规则源驱动。
4. **回单与反馈导出存在口径分叉**：`returned-batch` 与 `batch` 两条导出路径状态过滤不同，易造成业务方误解。

---

## 2. 端到端链路审计

### 2.1 标准链路

录单（order-parse）
→ 订单中心（orders）
→ 发货通知单（shipping-export）
→ 物流回单（return-receipt）
→ 客户反馈导出（feedback-export）
→ 导出记录（export-records）

### 2.2 当前关键断点

1. **发货通知单导出模式复杂**：preview/dispatch/persistence 组合多，用户难以理解副作用边界。
2. **回单确认对订单状态有追加逻辑**：同一订单多次确认会走 `partial_returned` 分支，业务理解门槛高。
3. **客户反馈导出路径存在“双入口口径”**：
   - `returned-batch` 只取 `returned`
   - `batch` 允许更宽状态集合（含已反馈/已完成）
4. **模板配置与列映射关系复杂**：有客户映射时优先映射，无映射时回退模板；用户易误以为“永远按原列名导出”。

## 3. 模块级深度分析

## 3.1 发货通知单（`/shipping-export`）

### 主要问题

1. **导出动作语义不够清晰**
   - UI 上“导出”可能同时触发派发、状态更新、导出记录、文件持久化。
2. **待导出/已导出双 Tab 的业务边界依赖后端聚合**
   - 用户不容易理解“为什么某发货方出现在已导出但仍有待处理单”。
3. **按钮禁用规则可解释性不足**
   - 缺少 `disabledReason` 可视化。

### 建议

- 在 UI 增加“本次操作副作用清单”（会不会改状态/扣库存/落记录）。
- 批量导出前展示 impact preview（预计影响订单数、状态变化数、重复复用数）。
- 发货通知单结果弹窗输出“模板来源 + 副作用摘要 + 可回滚信息”。

## 3.2 物流回单（`/return-receipt`）

### 主要问题

1. **导入容错高但可控性弱**
   - 接受多种列名与格式，降低门槛但提高误匹配风险。
2. **自动匹配/冲突/手工确认链路较长**
   - 缺少统一的“匹配置信度评分”展示，复核效率受影响。
3. **状态更新逻辑对用户不透明**
   - 确认回单后可能是 `returned` 或 `partial_returned`，用户不易预测。

### 建议

- 增加“匹配评分+命中规则”显示（快递单号/订单号+手机号等）。
- 确认前预览“将变更的订单状态与字段”。
- 对重复导入与重复确认增加幂等提示和可追溯原因。

## 3.3 客户反馈导出（`/feedback-export`）

### 主要问题

1. **状态口径有分叉风险**
   - `returned-batch` 与 `batch` 在筛选状态上的差异易导致“同客户不同导出结果”。
2. **映射优先级分支复杂**
   - 客户列映射 vs 模板映射回退，需要更明确可视化说明。
3. **导出结果解释不足**
   - 用户往往只看文件名，不知道本次列名来源是“客户映射”还是“模板”。

### 建议

- 前端强展示 `templateSource`：`column_mapping/default/explicit/first/mixed`。
- 同一页面统一“导出范围口径”提示（是否仅 returned）。
- 缺失客户映射时弹二次确认：明确“将按模板字段导出，非客户原列名”。

## 3.4 模板配置（`/templates`）

### 主要问题

1. **模板数据结构多形态（对象/数组/JSON字符串）**
   - 读写转换逻辑复杂，容易出现边缘 case。
2. **默认模板切换是隐式全局副作用**
   - 设置默认会取消同类型其他默认，缺少审计反馈。
3. **模板与目标绑定关系的可见性不足**
   - targetType/targetId/targetName 绑定历史不直观。

### 建议

- 统一模板存储协议（只保留一种规范格式）。
- 默认模板变更时输出“受影响目标”预览。
- 提供模板“生效路径追踪”：这次导出用了哪个模板、为什么。

---

## 4. 跨模块共性问题（全局视角）

1. **规则源分散**：状态机、模板优先级、映射优先级散落在多个页面和路由。
2. **双路径实现**：同一业务能力存在多个 API 入口，口径容易分叉。
3. **可用性与可执行性不一致**：按钮可点但后端拒绝的体验风险仍在。
4. **结果可解释性不足**：缺少“这次为什么是这个结果”的系统性说明。

## 6. 分阶段整改路线（四模块专项）

## 阶段A（P0）：口径统一与契约先行

1. 固化三套契约：
   - 状态机契约（哪些状态能触发哪些导出/回单动作）
   - 模板优先级契约（专属 > 默认 > 内置）
   - 导出口径契约（returned-only vs feedback-source）
2. 建立按钮矩阵：`visible/enabled/disabledReason`。

## 阶段B（P0）：动作服务化

1. shipping-export / return-receipt / feedback-export 的批量动作下沉服务层。
2. 页面只做编排，后端统一返回 `success/skipped/failed` 明细。

## 阶段C（P1）：模板治理与可解释性

1. 统一模板存储结构。
2. 新增“本次导出决策轨迹”（模板来源、映射来源、降级原因）。
3. 所有导出结果弹窗增加“操作副作用摘要”。

## 阶段D（P1）：回归与观测

1. 新增跨模块业务烟囱测试：录单→派发→回单→反馈导出→导出记录追溯。
2. 增加关键埋点：模板来源、匹配冲突率、回单确认失败率、导出回退率。

---

## 7. 验收标准

1. 同一订单在四模块动作后，状态流转可被状态机解释且一致。
2. 同一客户导出反馈单时，列名来源和数据来源可解释、可追溯。
3. 所有批量按钮都能给出明确禁用原因。
4. 双路径接口（如 returned-batch 与 batch）口径差异在 UI 可见且有说明。
5. 出现模板回退/匹配回退时，系统有明确提示与记录。

---

## 8. 风险提示

- 不建议先做 UI 大改；应先收敛规则与契约。
- 不建议先删旧路径；应先加观测并运行一段灰度。
- 不建议跨模块一次提交；应按“契约 -> 服务 -> 页面 -> 清理”顺序迭代。
