# 礼品订单管理系统 — AI执行交接指引（订单录入 / 订单管理）

> 版本：v1.0 | 更新日期：2026-05-01 | 适用对象：Claude Code / Gemini / 其他编程型大模型

## 1. 文档目的

本文档用于把以下两份方案转化为“可执行交接包”：

1. `docs/Product/AI订单录入模块深度分析与整改方案.md`
2. `docs/Product/订单管理模块深度分析与整改方案.md`

目标：让其他模型**少走弯路、少做大重构、少引入口径漂移**，并能稳定分阶段落地。

---

## 2. 给其他模型的最高优先级建议（先看）

1. **不要一次性重构**：先做“规则中心化/状态机中心化”，再做页面与路由拆分。
2. **先立契约再改代码**：先补测试（状态流转、匹配优先级、按钮可用性矩阵），再改实现。
3. **保证同一事实来源（SSOT）**：
   - 订单录入：匹配规则、列映射规则只保留一个来源。
   - 订单管理：状态流转、按钮可用性、后端校验使用同一配置。
4. **兼容逻辑要隔离**：snake/camel、新旧字段兼容放 adapter，不要散落在业务流程中。
5. **每个 PR 只做一个关注点**：例如“只收敛状态机”，不要混合“状态机+页面大改+API重写”。

---

## 3. 推荐执行顺序（强烈建议）

## 阶段 0：基线与保护（必须先做）

### 0.1 先冻结基线
- 跑现有验收命令，记录当前失败项（作为“既有问题”）
- 输出当前行为快照（特别是状态流转、按钮禁用逻辑）

### 0.2 先补最小保护测试
- 订单录入：
  - 列映射保存/复用契约测试
  - 匹配优先级契约测试
- 订单管理：
  - 状态机流转契约测试
  - 批量按钮可用性矩阵快照测试

**原则**：没有测试保护，先不要动核心流程。

## 阶段 1：规则/状态机中心化（低风险高收益）

### 1.1 订单录入
- 抽出统一 `match-policy`（或等价模块）
- 移除重复 `CHINESE_COLUMN_MAPPING` / `autoDetectColumnMapping` 私有实现
- 统一 `customer_id` 查询口径（旧字段仅兼容层读取）

### 1.2 订单管理
- 引入 `order-workflow-config`（或等价）
  - `allowedTransitions`
  - `actionsByStatus`
  - `editableFieldsByStatus`
  - `bulkActionRules`
- UI 与 API 共用同一规则源

## 阶段 2：动作服务化与页面瘦身

### 2.1 订单录入
- 提交逻辑统一入口（避免 page/hook 双实现）
- 映射保存结果可见化（成功/失败有提示，不只 console.warn）

### 2.2 订单管理
- 批量动作下沉到 service 层
- 将 `selectedOrders: Set<Order>` 改为 `Set<string>`
- 拆分 `/api/orders/route.ts`（先增量拆分，保留兼容代理）

## 阶段 3：兼容层隔离 + 清理重复

- 新建 adapter（order-parse / orders 各自 adapter）
- 清理页面与路由中的字段兼容分支
- 删除已弃用路径（必须有监控和回滚方案）

## 5. 常见失败模式（请提醒其他模型避免）

1. **先改 UI 后改规则**：视觉上“变好”，但底层仍分叉，问题更难查。
2. **只改前端不改后端校验**：按钮可点，API 拒绝，用户体验更差。
3. **拆路由不补契约测试**：短期通过，后续边界条件崩。
4. **把兼容逻辑继续塞业务层**：技术债换个位置继续增长。
5. **把多个阶段一次做完**：无法回滚，评审难通过。

## 6. PR 粒度建议（可直接照抄）

### PR-1（保护）
- 新增测试：状态机契约 + 匹配优先级 + 列映射复用 + 按钮矩阵
- 不改业务代码

### PR-2（规则中心化）
- 新增 `match-policy` / `order-workflow-config`
- 让旧代码调用新配置
- 行为保持不变

### PR-3（动作服务化）
- 批量动作下沉 service
- UI 改为统一 action executor

### PR-4（选择集与页面瘦身）
- `Set<Order>` -> `Set<string>`
- page.tsx 只保留编排入口

### PR-5（兼容层隔离）
- adapter 落地
- 清理重复和历史兜底分支

---

## 7. 验收标准（交给任何模型都适用）

1. 同一状态下：UI可见动作 == API可执行动作。
2. 同一客户同模板 Excel：映射可自动复用；列名变化时给出明确提示。
3. 客户反馈导出：有映射时按原始列名导出；无映射时必须提示使用模板回退。
4. 批量动作重复执行：不产生脏数据（幂等）。
5. 分页/刷新后：勾选状态稳定不丢失。

## 9. 最后建议（给你）

如果你准备把任务交给 Claude Code 或 Gemini，建议你把任务拆成“短 sprint 指令”：

- Sprint A：只做 PR-1 + PR-2
- Sprint B：只做 PR-3
- Sprint C：只做 PR-4 + PR-5

每个 sprint 结束都要求：
1) 变更摘要；2) 风险；3) 回滚方案；4) 验证命令与实际输出。

这样可以明显降低 AI 一次性大改带来的不可控风险。
