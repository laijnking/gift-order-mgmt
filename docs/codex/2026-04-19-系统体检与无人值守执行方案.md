## 2. 总体判断

### 2.1 总体评级

- 产品完成度：`中`
- 工程稳定性：`中偏低`
- 数据一致性：`低`
- 自动化测试完备度：`低`
- 可无人值守改造性：`高`

### 2.2 结论摘要

系统目前已经具备 MVP 业务骨架，但尚未达到“稳定可放手迭代”的状态。最大的问题不是单点 bug，而是以下四类结构性风险：

1. **同一业务存在多条实现路径，且行为不一致**
2. **前端页面约定、API 约定、数据库 schema 之间存在漂移**
3. **角色、用户、客户、订单之间的关联是“部分真实、部分冗余、部分兼容逻辑”混合态**
4. **导入导出链路缺少严格模板、校验、回放与验收闭环**

换句话说，现在系统“能跑”，但还不够“可信”。

## 4. 高优先级问题清单

### P0-1 订单录入时，客户带出业务员 / 跟单员依赖姓名硬匹配，且角色过滤不一致

**现象**

- 选择客户后，页面通过客户档案里的姓名去用户列表中做精确匹配。
- 业务员下拉只展示 `salesperson`。
- 跟单员下拉只展示 `order_taker`。

**代码位置**

- 客户选择后通过姓名匹配用户：[src/app/(app)/order-parse/page.tsx:1344](/root/project/gift-order-mgmt/src/app/(app)/order-parse/page.tsx#L1344)
- 业务员下拉仅过滤 `salesperson`：[src/app/(app)/order-parse/page.tsx:1412](/root/project/gift-order-mgmt/src/app/(app)/order-parse/page.tsx#L1412)
- 跟单员下拉仅过滤 `order_taker`：[src/app/(app)/order-parse/page.tsx:1448](/root/project/gift-order-mgmt/src/app/(app)/order-parse/page.tsx#L1448)
- 客户管理页对角色的过滤规则却是另一套：[src/app/(app)/customers/page.tsx:157](/root/project/gift-order-mgmt/src/app/(app)/customers/page.tsx#L157)

**影响**

- 客户档案已经绑定了业务员 / 跟单员，录单页却可能带不出来
- 一旦用户真实姓名带括号、别名、旧名，联动就失效
- 销售主管、跟单员等角色在不同页面能选 / 不能选，行为不一致

**根因**

- 系统没有统一使用 `sales_user_id / operator_user_id` 作为唯一关联
- 页面之间对角色编码的理解不一致

**修复策略**

1. 订单录入页优先使用客户档案中的 `salesUserId / operatorUserId`
2. 若仅有姓名冗余，才进入兼容匹配
3. 建立统一角色分组函数：销售岗、跟单岗、管理岗
4. 客户档案、订单录入、订单导入三处共用同一套角色筛选

**验收标准**

- 选择客户后，业务员 / 跟单员稳定按 ID 回填
- `salesperson`、`sales_manager`、`operator`、`order_taker` 的展示行为一致

### P0-2 订单录入存在两套解析引擎，文本录入与 Excel 导入能力模型不一致

**现象**

- 文本录入接口虽然读取 `agent_configs`，但实际仍使用本地规则解析。
- Excel 录入走另一套增强版解析、商品匹配和供应商匹配逻辑。

**代码位置**

- 文本录入实际使用规则解析：[src/app/api/order-parse/route.ts:33](/root/project/gift-order-mgmt/src/app/api/order-parse/route.ts#L33)
- 规则解析实现：[src/app/api/order-parse/route.ts:87](/root/project/gift-order-mgmt/src/app/api/order-parse/route.ts#L87)
- Excel 解析入口与自动映射：[src/app/api/order-parse/excel/route.ts:4](/root/project/gift-order-mgmt/src/app/api/order-parse/excel/route.ts#L4)

**影响**

- 同一客户、同一订单，文本录入与 Excel 导入得到的结果粒度不同
- 上游解析结果字段不统一，后续保存订单、匹配商品、匹配供应商时容易出现兼容分支

**根因**

- 解析领域模型没有统一
- “文本解析、Excel 映射、商品匹配、供应商匹配”被拆散在多层实现里

**修复策略**

1. 定义统一的 `ParsedOrderDraft` 领域对象
2. 文本录入与 Excel 导入都产出同一结构
3. 商品匹配、供应商匹配、客户联动改为后置增强步骤
4. 明确当前版本是否启用 LLM；未启用则移除“Agent 默认配置”假象

**验收标准**

- 文本 / Excel 都输出同一订单草稿结构
- 同一字段不会再出现驼峰、下划线、旧字段三套混杂

### P0-3 Excel 字段映射链路可保存，但不够可信，且接口语义混乱

**现象**

- 页面支持自动识别表头、保存映射、恢复历史版本
- 但保存的是客户级映射快照，缺少模板来源、导入批次、冲突校验、字段完整性校验
- 历史版本查询使用 `PUT` 语义不合理

**代码位置**

- 保存 / 获取字段映射：[src/app/api/column-mappings/route.ts:4](/root/project/gift-order-mgmt/src/app/api/column-mappings/route.ts#L4)
- 历史版本查询使用 `PUT`：[src/app/api/column-mappings/route.ts:104](/root/project/gift-order-mgmt/src/app/api/column-mappings/route.ts#L104)
- 订单录入页加载 / 保存映射：[src/app/(app)/order-parse/page.tsx:633](/root/project/gift-order-mgmt/src/app/(app)/order-parse/page.tsx#L633)

**影响**

- 映射配置“能保存”，但无法证明该映射适用于哪个模板版本
- 出错后难以复盘到底是表头变化、映射错误还是客户模板变化

**修复策略**

1. 映射配置升级为“客户 + 文件模板签名 + 表头指纹”三元组
2. 历史接口改为 `GET /api/column-mappings/history`
3. 保存前加入必填字段校验与冲突提示
4. 导入结果页显示“本次自动映射命中率 / 未识别字段 / 冲突字段”

**验收标准**

- 任一 Excel 导入都能追溯到具体映射版本与模板签名
- 保存映射不再只是“当前客户的一份快照”

### P0-6 发货通知导出、订单派发、库存扣减之间职责边界不清

**现象**

- 存在多个导出 / 派发入口
- 部分接口在导出时直接触发库存扣减与派发记录写入
- 业务动作、副作用、导出行为耦合较深

**代码位置**

- 订单派发与库存扣减：[src/app/api/export/route.ts](/root/project/gift-order-mgmt/src/app/api/export/route.ts)
- 发货通知批量导出：[src/app/api/shipping-exports/batch/route.ts](/root/project/gift-order-mgmt/src/app/api/shipping-exports/batch/route.ts)

**影响**

- “只是导出一份表”与“真正完成派发”之间边界不清
- 容易出现重复扣库存、重复写成本、重复建派发记录

**修复策略**

1. 把业务动作拆成三个显式步骤：预览、确认派发、导出文件
2. 库存扣减与成本写入只允许发生在“确认派发”
3. 文件导出是派发结果的视图，不再驱动业务状态

**验收标准**

- 同一订单无论点多少次导出，都不会重复扣库存

**现象**

- 但操作按钮、后端 API 仍未统一按权限校验

3. 为系统设置、导入导出、批量操作补按钮级权限

**验收标准**

- 菜单、页面、按钮、API 四层权限一致

## 5. 中优先级问题清单

### P1-1 客户管理、用户管理、订单录入三处角色编码体系仍不完全统一

- 客户管理页接受 `sales / salesperson / salesman / sales_manager`
- 订单录入页只认 `salesperson`
- 跟单岗在不同页面分别出现 `operator / order_taker`

建议统一角色字典与角色分组工具。

### P1-2 回单导入匹配策略偏弱，缺少去重与人工复核闭环

- 当前按匹配码、订单号+电话、订单号+姓名、订单号、电话+商品依次尝试
- 但缺少“重复单号拦截”“同单重复导入保护”“待人工处理池”

建议建立 `matched / ambiguous / duplicate / unmatched` 四态。

### P1-3 预警规则与预警记录之间，规则开关对真实业务动作影响有限

- 页面能维护规则
- 但诸如尾货预警仍直接在业务代码里写入记录

建议将“是否触发、阈值、通知方式”从硬编码迁移到规则执行器。

### P1-4 报表可信度依赖上游状态流，目前状态定义仍不够收敛

- 状态枚举、导出状态、回单状态、反馈状态存在偏差
- 导致报表口径可能随模块实现而变化

建议先统一状态机，再校准统计 SQL / API。

## 6. 低优先级问题清单

- 文档中仍保留旧平台、旧地址、旧说明
- 多个页面存在 Excel 模板下载、导入提示、默认值说明未统一
- 部分 API 仍有 TODO 输出、`current_user` 占位、mock 逻辑

## 8. 无人值守执行规则
