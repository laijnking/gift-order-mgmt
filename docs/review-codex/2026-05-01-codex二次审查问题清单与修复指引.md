## 1. 复审结论总览

## 1.1 已明显解决的问题

1. **列映射规则去重（阶段A部分完成）**
   - `order-parse/excel/route.ts` 已依赖 `column-mapping-rules`，未再发现路由内独立维护的大型映射常量双轨逻辑。

2. **录单提交流程去重（阶段B部分完成）**
   - `order-parse/page.tsx` 已通过 `useOrderSubmit` 统一提交 `/api/orders` 与异步保存映射流程。

3. **订单页勾选稳定性改造已落地**
   - 已采用 `selectedOrderIds` 思路，避免对象引用集合导致的分页错位问题。

## 1.2 未解决或仍有偏差的问题（重点）

1. **商品匹配优先级口径冲突（高优先级）**
   - 代码当前执行口径：`code > barcode > spec > name`；
   - 项目业务文档口径：`spec > code > barcode > name`；
   - 这是会直接影响匹配结果与成本推荐的核心规则冲突。

2. **`customer_id` 收敛未完成（高优先级）**
   - 虽然 `product-match` 与 `order-parse/excel` 已改为 `customer_id` 查询映射；
   - 但代码库中仍有大量接口以 `customer_code` 为主查询口径（如 `order-parse/route.ts`、`orders/counts`、`column-mappings` 等），迁移处于长期中间态。

3. **`/api/orders/route.ts` 巨型路由问题仍在（高优先级）**
   - 文件约 1400+ 行，查询/创建/更新/删除/导出相关逻辑仍高耦合，未完成动作服务化与职责拆分。

4. **`feedbacked` 状态语义未收敛（中优先级）**
   - 仍作为真实状态参与页面链路与状态定义，和“统计聚合口径”的认知边界仍可能混淆。

5. **全量验收失败：基础依赖缺失（阻断项）**
   - `pnpm check:unattended-acceptance` 失败，`ts-check` 报错：`Cannot find module 'nodemailer'`。
   - 在该阻断项修复前，任何“已完成整改”的结论都不成立。

## 1.3 深度探究新增问题

1. **`match-policy.ts` 字段映射语义可疑**
   - `SKU_MAPPING_FIELDS.spec` 当前映射到 `customer_product_name`，需业务确认是否设计如此；若非刻意设计，会导致规格匹配偏差。

2. **策略 SSOT 宣言与全仓调用现实尚未闭环**
   - 录单 Excel 路由已接入策略模块，但其他匹配入口（如 `/api/match`）尚未完全共享同一套策略实现，存在再次漂移风险。

## 2. 给“其他大模型”的强制执行指引（按顺序）

> 以下为可直接执行的修复任务单。严禁跳步。

### Step 1（阻断项）先恢复无人值守验收通道

**目标**：让 `pnpm check:unattended-acceptance` 至少能跑完，不被基础依赖中断。

1. 修复 `nodemailer` 缺失问题：
   - 检查 `package.json` 是否缺依赖与类型；
   - 安装或补齐正确依赖；
   - 如项目采用 SMTP 抽象层，确保导入路径与 tsconfig moduleResolution 一致。
2. 运行：
   - `pnpm install`
   - `pnpm ts-check`
   - `pnpm check:unattended-acceptance`
3. 输出证据：完整命令与原始输出摘要。

**验收标准**：不再出现 `Cannot find module 'nodemailer'`。

---

### Step 2 统一“商品匹配优先级”口径并固化契约测试

**目标**：消除“文档一套、代码一套”。

1. 先与业务口径确认最终顺序（推荐按 AGENTS 业务字典：`spec > code > barcode > name`）。
2. 同步修改：
   - `src/lib/order-parse/match-policy.ts`
   - `src/app/api/order-parse/excel/route.ts`（若仍有 if/else 顺序依赖）
   - 其他匹配入口（含 `/api/match`、`/api/product-match`）
3. 新增/补齐测试：
   - 至少覆盖四种输入冲突场景，验证命中顺序。
4. 更新文档：
   - `docs/Product/模块功能说明.md`
   - `docs/Product/模块交互说明.md`（如果接口行为受影响）
   - 文档顶部追加版本记录。

**验收标准**：同一输入在不同匹配入口返回一致的 `matchType` 与 `productId`。

---

### Step 3 完成 `customer_id` 口径收敛（保留兼容层，不再扩散）

**目标**：实现“新逻辑只写 `customer_id`，`customer_code` 仅兼容读取”。

1. 盘点所有 `customer_code` 查询入口（先用 `rg -n "customer_code" src/app/api src/lib`）。
2. 分级改造：
   - P0：匹配、录单、统计主链路；
   - P1：报表、导出、历史记录；
   - P2：仅管理后台检索类接口。
3. 为每个改造接口补兼容逻辑：
   - 入参可带 code，但服务内部先解析到 id；
   - 数据库查询优先 id；
   - 保留旧字段仅用于 fallback。
4. 增加静态守卫：
   - 在 lint 或脚本中阻断“新代码直接按 `product_mappings.customer_code` 查询”。

**验收标准**：新增代码不再引入新的 `customer_code` 主查询；回归不破坏旧数据读取。

---

### Step 4 拆分 `/api/orders/route.ts`，落地动作服务层

**目标**：降低巨型路由回归风险。

1. 拆分建议：
   - `GET /api/orders`（查询）
   - `POST /api/orders`（创建）
   - `PATCH /api/orders/actions/*`（批量动作）
   - `DELETE /api/orders`（删除）
2. 抽取共用层：
   - `src/lib/order-workflow-config.ts`：状态机与允许动作
   - `src/lib/orders-action-service.ts`：批量动作执行与结果聚合
3. 统一返回结构：
   - `{ successCount, skippedCount, failed: [{id, reason}] }`
4. 保留向后兼容：
   - 旧调用路径过渡期内可代理到新服务。

**验收标准**：单一改动不再触发整个 orders 路由大面积回归。

---

### Step 5 状态语义收敛（`feedbacked`）

**目标**：统一“业务状态节点”和“统计聚合口径”边界。

1. 明确决策：
   - A：保留 `feedbacked` 真实状态；或
   - B：改为统计聚合概念，不再作为独立流转节点。
2. 同步改造：
   - `src/lib/order-status.ts`
   - `/orders` 页面按钮与批量动作规则
   - 相关报表与导出状态筛选逻辑
3. 补测试：状态可流转矩阵测试。
4. 更新文档（模块功能 + 交互说明 + 数据字典如涉及字段枚举）。

**验收标准**：UI 可操作动作、API 校验、报表统计三方口径一致。
