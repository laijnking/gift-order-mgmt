Section · 01

订单核心

订单录入 → 订单中心 → 状态管理 → 批量派发
每天 80% 的操作都在这四个页面里

好，我们现在进入第一部分——订单核心。这四个功能是大家每天打开系统就会用到的。我们从订单列表开始讲起。

Orders · 订单中心

8 种状态，一条流转线

待派发 pending → 已派发 assigned → 通知发货 notified → 部分回单 partial_returned
部分回单 → 已回单 returned → 已反馈 feedbacked → 已完成 completed ← 已取消 cancelled

🔍 订单搜索与筛选

  • 按订单号 / 收货人 / 手机号搜索
  • 按状态 / 客户 / 时间范围 / 导入批次筛选
  • 支持批量选择后统一派发
  • 按导入批次查看复盘摘要

📌 重要区别

  • assigned = 内部派单（后台操作）
  • notified = 外部通知（已导出通知单）
  • feedbacked = 独立状态，已反馈但未归档
  • 订单号：order_no（客户原始） / sys_order_no（系统唯一）
订单状态流转是系统最核心的概念。特别注意 assigned 和 notified 的区别——assigned 是内部派了单但还没通知发货方，notified 才是真正导出了通知单。另外 feedbacked 是一个独立状态，表示已经反馈给客户了但还没归档。这些状态区分清楚，日常操作才不会乱。

Shipping Export · 发货通知单

三种派发模式，按需选择

👁️ 预览模式

preview

  • 仅预览内容，不做任何修改
  • 不派发、不扣库存
  • 不写入任何记录
  • 适合检查核对

📦 仅派发

dispatch_only

  • 派发订单 + 扣减库存
  • 写入派发记录和成本记录
  • 不写导出记录
  • 不生成持久化文件

📁 派发并留痕

dispatch_with_persistence

  • 派发 + 扣库存 + 写记录
  • 生成导出记录
  • 文件持久化（本地/S3）
  • 完整留痕，可追溯

🔑 发货方商品编码取数规则

优先取发货方商品档案中匹配的编码：派发时，系统通过 product_mappings 表按 发货方 ID + 商品 ID 查找该发货方为这个商品配置的自定义编码（supplier_product_code）。
匹配不到则默认使用系统商品编码：若该发货方未为此商品配置映射，则回退使用系统 stock.product_code。
⚡ 匹配结果以快照形式写入 dispatch_items，后续导出不再跨表查询，保证导出效率和数据一致性。

其他关键机制： 派发幂等 — 同一订单重复派发复用已有记录，不重复扣库存 | 发货方专属模板通过 targetType + targetId 精确匹配

这页最重要的是两个规则。第一，发货方商品编码的取数逻辑——优先取发货方自己配置的商品编码，没有配就用系统默认的。这样既能满足不同发货方对同一个商品有不同叫法的需求，又不会因为没有配置而导不出数据。第二，三种派发模式的区别。日常建议先用 preview 预览，确认没问题再正式派发。

Return Receipt · 物流回单

四级匹配，自动 + 人工兜底

上传回单 Excel，系统按优先级自动匹配订单。单一匹配自动确认，多候选进入复核池。

优先级匹配规则精确度场景
#1系统订单号 sys_order_no 精确匹配⭐⭐⭐⭐⭐系统中已有唯一订单号
#2客户订单号 order_no 精确匹配⭐⭐⭐⭐按客户原始订单号查找
#3客户订单号 + 收货人手机号⭐⭐⭐双重校验，减少误匹配
#4收货人手机号 + 商品名称⭐⭐兜底匹配，无订单号时使用

匹配结果处理

  • auto_matched — 唯一匹配，自动确认
  • conflict — 多候选，进入复核池人工处理
  • manual_matched — 人工选择后确认
  • unmatched — 无匹配，需手动关联

回单确认后的动作

  • 补全 order_cost_history 的快递信息
  • 更新订单状态（partial_returned / returned）
  • 部分回单：物流信息用 ";" 隔开追加
  • 快递公司和单号均为空 → 仅关联，不更新状态
回单匹配的核心逻辑是四级递进。系统订单号最准但需要发货方在回单里填了 sys_order_no。大多数情况是走第二或第三级——用客户订单号匹配。如果匹配出来有多个候选，订单会进复核池，需要人工确认选哪个。部分回单的场景很常见，比如一个订单 10 件商品，发货方分两批发货，系统会用分号把两个快递单号拼在一起。

Feedback Export · 客户反馈 + 导出记录

客户用什么列名导入，就用什么列名导出

📤 客户反馈导出

  • 按 feedback_export_headers 还原客户原始列名
  • 物流信息自动追加在末列
  • column_order 字段保证列顺序一致
  • 支持二次导出、导出历史查询

📁 导出记录管理

  • 查看所有导出历史（发货通知 + 客户反馈）
  • 整批 ZIP 打包下载
  • 明细单独重新生成并下载
  • 模板来源标注：supplier / default / mixed

文件下载策略

文件状态下载方式
本地持久化文件存在直接下载本地文件
对象存储（S3）文件存在返回预签名直链（900 秒有效）
未持久化按需重新生成 ZIP / Excel
客户反馈导出有个很重要的设计——客户用什么列名导入的 Excel，导出时就还原成什么列名。这样客户收到的反馈表和当初提供的订单表格式完全一致，不用重新对列。导出记录模块是所有导出的历史存档，可以重新下载，也可以单独重新生成某一条。

Section · 02

库存与预警

库存查询 · 变更追踪 · 价格历史 · 预警监控
保障供应链不中断的日常巡检工具

接下来是库存和预警。这两个模块是保障日常运营不翻车的关键——库存不足会导致派发失败，预警不及时可能导致订单延误。每天上班花 5 分钟看一眼预警中心是个好习惯。

Stocks · 库存管理

6 种变更类型，全链路可追溯

手动调整 manual
批量导入 import
订单扣减 order
系统校正 adjust
退货入库 return
调拨 transfer

📊 库存查询

  • 按商品 × 发货方查询实时库存
  • 库存 ≤ 2 台触发尾货预警
  • 库存不足 / 负数禁止派发
  • 支持批量导入更新库存

📜 变更追溯

  • stock_versions — 每次库存变更快照
  • price_history — 价格调整历史
  • 派发时自动写入 stock_versions（change_type: "order"）
  • 变更记录包含 quantity_before / quantity_after

⚠️ 库存扣减流程：派发 → 更新 stocks.quantity → 写入 stock_versions（change_type: "order"）→ 库存 ≤ 2 触发预警

库存变更有六种类型，其中最需要关注的是订单扣减——它是派发时自动触发的，不需要手动操作。每次库存变更都会在 stock_versions 表里留一条记录，包含变更前后的数量，出问题了可以追溯到是哪一步。

Alerts · 预警中心

三阶段执行器：新增 → 复用 → 自动关闭

📉 库存不足预警

≤ 2 台

规则编码：LOW_STOCK_ALERT
库存低于阈值时触发

⏰ 待派发超时预警

> N 小时

规则编码：ORDER_TIMEOUT_ALERT
待派发超过配置时长触发

🚚 回单超时预警

> N 小时

规则编码：RETURN_DELAY_ALERT
派发后超过时长未回单触发

1

新增

满足规则条件
且无未处理预警
→ 创建 alert_records

2

复用

满足规则条件
但已有未处理预警
→ 跳过，避免重复

3

自动关闭

不再满足规则条件
但存在未处理预警
→ is_resolved = true

预警的三阶段执行器设计思路是"不重复报警、自动恢复"。同一个问题已经有了未处理的预警记录就不会再重复创建；问题自己解决了（比如库存补上了），系统会自动把预警关闭。日常只需要关注处于第一阶段——新增出来的预警。

Section · 03

档案管理

客户档案 · 发货方管理 · 商品管理 · SKU 映射
基础数据维护，一次配置长期受益

最后一部分是档案管理和报表。这些模块操作频率不高——大多数是第一次配置好之后就不怎么动了——但数据准确与否直接影响订单录入和派发的效率。

Archive · 客户 + 发货方

两个核心档案，管理上下游

👥 客户管理

  • 客户档案创建、编辑、查询
  • 分配业务员（salesperson）和跟单员（operator）
  • 批量交接 — 一批客户从一个业务员转给另一个
  • 数据权限：业务员只能看自己的客户（数据范围: self/department/all）

🏭 发货方管理

  • 5 种类型：传统发货方 / 京东 / 拼多多 / 自有仓 / 第三方代发
  • 配置快递限制（省份不兼容不可派发）
  • 管理发货方商品编码
  • shippers 表是唯一数据源（suppliers 表已废弃）
类型值说明发货方式典型场景
supplier传统发货方download（下载通知单）常规合作工厂
jd京东jd（API 对接京东仓）京东一件代发
pdd拼多多pdd（API 对接拼多多仓）拼多多一件代发
self自有仓self自建仓库发货
third_party第三方代发download外部代发服务商
客户管理里有一个实用功能——批量交接。比如一个业务员离职了，可以一键把他名下的所有客户转给另一个业务员。发货方管理支持五种类型，最常见的是传统发货方和自有仓。京东和拼多多是 API 对接的，配置好后可以实现自动化。

商品管理和 SKU 映射是订单录入的前置条件——如果商品档案没建好，导入时匹配率就会很低。数据报表有四类，运营最常用的是订单统计和销售业绩。注意报表里的统计有两种口径：严格按单个状态计数，和回单阶段的四种状态合并统计。
